# robinreinecke.de

Personal site + blog. Astro, Tailwind, shadcn-style components ported as plain `.astro` files, self-hosted as a Docker container.

## Development

```sh
pnpm install
pnpm dev          # http://localhost:4321
pnpm check        # typecheck (astro check)
pnpm build        # static output to ./dist

pnpm lint         # ESLint
pnpm lint:fix     # ESLint with autofix
pnpm format       # Prettier write
pnpm format:check # Prettier check (used in CI)
```

Formatting is handled by Prettier (with Astro and Tailwind class-sorting plugins); linting by
ESLint (flat config, TypeScript + Astro). CI runs `format:check`, `lint`, `check`, and `build` on
every push and pull request. If you use VS Code, the recommended extensions and format-on-save are
already configured in `.vscode/`.

## Content

Blog posts live in `src/content/blog/*.mdx` (or `.md`). Frontmatter schema is defined in
`src/content.config.ts`. Set `draft: true` while iterating on a post locally. Draft posts never
build to production and never trigger the social-draft pipeline below.

## SEO & social previews

Every page emits a full meta set from `src/components/site/Seo.astro` (wired through
`BaseLayout`): description, canonical URL, robots, Open Graph, and Twitter `summary_large_image`
cards. Structured data (JSON-LD) is injected per page: `WebSite` + `Person` on the home page and
`BlogPosting` on each post. A `sitemap-index.xml` is generated by `@astrojs/sitemap` and pointed to
from `public/robots.txt`.

Social preview images (1200x630) are generated at build time by `astro-og-canvas` from
`src/pages/og/[...route].ts` — one branded card per blog post (`/og/blog/<slug>.png`) plus a
site-wide default (`/og/default.png`). Fonts (`src/assets/fonts/*.ttf`) and the logo
(`src/assets/og-logo.png`) are bundled in the repo so the build never needs network access. To
change the card style, edit `getImageOptions` in that route.

## Deployment

```sh
docker build -t robinreinecke-site:latest .
```

## Social cross-posting

On every push to `main` that adds/modifies a non-draft post, `.github/workflows/social-draft.yml`
generates LinkedIn copy and an unpublished dev.to draft (requires the `DEVTO_API_KEY` repo
secret), then opens a GitHub Issue with everything ready to review. Nothing is posted
automatically — LinkedIn and X are copy-paste from the issue, and the dev.to draft only goes live
when you manually run the `social-publish.yml` workflow (`workflow_dispatch`) with the post's slug.
